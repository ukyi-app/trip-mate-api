## 1. 제품 개요

### 1.1 제품 설명

본 서비스는 친구들과 해외여행 중 발생하는 지출을 기록하고, 여행 종료 시 누가 누구에게 얼마를 보내야 하는지 자동으로 정리해주는 모바일 웹 기반 그룹 정산 서비스다.

초기에는 한국 사용자를 주요 대상으로 하며, 최종 송금 기준 통화는 기본적으로 KRW로 설정한다. 다만 여행지는 일본에 한정하지 않고, 사용자가 여행 국가와 현지 통화를 선택할 수 있도록 한다.

사용자는 Google 로그인을 통해 인증하며, 여행방은 어드민이 초대한 Google 계정만 접근할 수 있다. 지출은 현지 통화 금액과 정산 통화 금액을 모두 저장하며, 정산 통화 금액은 지출 사용 일자 기준 환율을 적용한다.

AI 기능은 대화형 챗봇이 아니라 지출 입력을 돕는 보조 기능으로 제공한다. 사용자는 자연어 문장, 카드 승인 문자, 영수증 사진, 카드 앱 캡처 등을 통해 지출 정보를 입력할 수 있으며, AI는 이를 분석해 지출 후보를 생성한다. 모든 AI 분석 결과는 사용자가 확인 및 수정한 뒤 저장한다.

### 1.2 한 줄 설명

친구들과 해외여행 중 발생한 현지 통화/원화 지출을 안전하게 기록하고, 최종 정산을 자동으로 계산해주는 초대형 여행 정산 앱.

### 1.3 문제 정의

친구들과 해외여행을 가면 식비, 교통비, 숙소비, 편의점 구매, 쇼핑, 입장권, 택시비 등 여러 지출이 계속 발생한다. 하지만 실제 여행 중에는 다음 문제가 반복된다.

- 누가 얼마를 냈는지 기억하기 어렵다.
- 현지 통화와 원화를 계속 계산해야 한다.
- 국가마다 통화, 소수점 단위, 결제 방식, 영수증 형식이 다르다.
- 지출 시점마다 환율이 달라질 수 있다.
- 카드 결제는 실제 원화 청구 금액이 별도로 존재할 수 있다.
- 모든 친구가 같은 지출에 참여하지 않는 경우가 많다.
- 여행 마지막 날 정산하려고 하면 기록이 누락된다.
- 카드 문자, 영수증, 메모, 계산기, 환율 앱을 오가야 해서 번거롭다.
- 링크만으로 접근 가능한 구조는 지출 데이터 보안과 신뢰성이 낮다.

### 1.4 제품 목표

여행 중 발생하는 지출을 빠르게 기록하고, 여행 종료 시 현지 통화 기준 정산과 정산 통화 기준 정산을 분리하여 신뢰 가능한 최종 정산 결과를 제공한다.

## 2. 제품 확장 방향

### 2.1 기존 일본 중심 구조의 문제

기존 PRD는 일본여행을 기준으로 `JPY`와 `KRW`를 중심에 두고 있었다. 하지만 서비스가 다른 국가로 확장되려면 특정 국가나 통화에 종속된 구조를 피해야 한다.

따라서 모든 금액, 환율, 영수증 분석, 정산 문구는 다음 개념으로 일반화한다.

| 기존 개념 | 개선 개념 |
|---|---|
| 일본여행 | 해외여행 |
| JPY | 현지 통화 |
| KRW | 정산 통화 |
| 엔화 기준 정산 | 현지 통화 기준 정산 |
| 원화 기준 정산 | 정산 통화 기준 정산 |
| 일본 영수증 | 해외 영수증 |
| 일본 카드 결제 | 해외 카드 결제 |

### 2.2 기본 타깃

서비스는 다국가 여행을 지원하되, 초기 타깃은 한국 사용자로 둔다.

기본값은 다음과 같다.

- 기본 정산 통화: KRW
- 기본 언어: 한국어
- 기본 인증: Google 로그인
- 기본 여행 형태: 친구 그룹 해외여행
- 기본 정산 방식: 정산 통화 기준 송금

## 3. 타깃 사용자

### 3.1 Primary User

한국인 20~30대 해외여행 그룹 여행자

### 3.2 사용 상황

- 일본, 대만, 태국, 베트남, 미국, 유럽 등 여러 국가 여행
- 모바일 웹으로 사용
- 3~6명 정도의 소규모 친구 그룹
- 현금, 카드, 교통카드, 간편결제 등 결제자가 매번 달라짐
- 모든 사람이 같은 비용을 나누지 않을 수 있음
- 최종 정산은 한국 원화 송금으로 처리하는 경우가 많음

### 3.3 대표 페르소나

#### 페르소나 A: 어드민/총무형 사용자

- 여행방을 만들고 친구들을 초대한다.
- 친구들이 “나중에 정산해줘”라고 맡기는 경우가 많다.
- 누락 없이 정확하게 정산하고 싶다.
- 누가 접근할 수 있는지 직접 통제하고 싶다.
- 여러 국가 여행에서도 같은 서비스로 정산하고 싶다.

#### 페르소나 B: 일반 멤버

- 복잡한 앱 사용을 싫어한다.
- 초대받은 Google 계정으로 로그인해서 바로 쓰고 싶다.
- 본인이 낸 지출을 빠르게 입력하고 싶다.
- 최종적으로 얼마를 보내야 하는지만 명확히 알고 싶다.

#### 페르소나 C: 기록형 사용자

- 영수증, 카드 내역, 날짜, 카테고리까지 남기고 싶다.
- 현지 통화 기준 지출과 정산 통화 기준 지출을 모두 보고 싶다.
- 여행 후 지출 내역을 회고하고 싶다.

## 6. 여행방 생성 정책

### 6.1 여행방 기본 정보

여행방 생성 시 다음 정보를 입력한다.

- 여행 이름
- 여행 시작일
- 여행 종료일
- 여행 국가
- 여행 시간대 (현지 TZ — 환율 일자 산출 기준, §14.2)
- 기본 현지 통화
- 정산 통화
- 어드민 표시 이름

### 6.2 여행 국가

여행 국가는 하나 이상 선택할 수 있다.

MVP에서는 단일 국가 여행을 우선 지원한다. 다중 국가 여행은 Phase 3 이후 확장 기능으로 둔다.

예시:

- 일본
- 대만
- 태국
- 베트남
- 미국
- 프랑스
- 이탈리아
- 영국

### 6.3 기본 현지 통화

여행 국가를 선택하면 기본 현지 통화를 추천한다.

예시:

| 국가 | 기본 현지 통화 |
|---|---|
| 일본 | JPY |
| 대만 | TWD |
| 태국 | THB |
| 베트남 | VND |
| 미국 | USD |
| 유로존 | EUR |
| 영국 | GBP |

사용자는 추천된 통화를 변경할 수 있다.

### 6.4 정산 통화

정산 통화는 친구들끼리 실제로 돈을 주고받을 기준 통화다.

초기 기본값은 KRW로 설정한다.

예시:

- KRW
- USD
- JPY
- EUR

한국 사용자를 대상으로 하는 MVP에서는 KRW 정산을 기본으로 한다.

## 7. 인증 정책

### 7.1 기본 인증 방식

본 서비스는 Google 로그인을 기본 인증 방식으로 사용한다.

모든 사용자는 여행방 생성 또는 참여 전에 Google 계정으로 로그인해야 한다.

### 7.2 인증이 필요한 작업

- 여행방 생성
- 여행방 참여
- 여행방 조회
- 지출 추가
- 지출 수정
- 지출 삭제
- 정산 결과 조회
- 멤버 초대
- 여행방 설정 수정
- 정산 확정
- 어드민 양도
- 여행방 삭제

### 7.3 사용자 식별 기준

사용자는 Google 계정을 통해 식별한다.

저장하는 사용자 정보는 다음과 같다.

- Google 계정 고유 ID
- 이메일
- 이름
- 프로필 이미지

사용자의 안정적 식별자는 Google 계정 고유 ID(google_account_id, Google sub)이며, 이메일은 변동 가능한 속성으로 취급한다.

여행방 접근 권한은 다음 두 단계로 판단한다.

- 참여(join) 시점: 로그인 사용자의 정규화 이메일이 초대 이메일과 일치하는지로 멤버십을 확정한다. (정규화 규칙은 8.5) Google에서 `email_verified=true`로 검증된 이메일만 초대 매칭에 사용하며, 미검증 이메일 로그인은 참여를 거부한다.
- 참여 이후: trip_members의 user_id(멤버십)로 권한을 판단한다. 이후 Google 이메일이 변경되어도 권한에 영향을 주지 않는다.

## 8. 어드민 및 초대 정책

### 8.1 어드민 정의

여행방을 생성한 사용자는 자동으로 해당 여행방의 어드민이 된다.

어드민은 여행방의 멤버, 설정, 권한, 정산 상태를 관리할 수 있다.

여행방에는 항상 active 어드민이 정확히 1명 존재한다. 어드민은 다른 참여 완료 멤버에게 어드민 권한을 양도할 수 있으며, 양도 시 기존 어드민은 일반 멤버가 된다. (승계 정책은 9.5)

### 8.2 초대 방식

여행방은 공개 입장 방식이 아니라 이메일 기반 초대 방식을 사용한다.

어드민은 친구의 이름과 Google 계정 이메일을 입력하여 멤버를 초대한다.

예시:

```txt
민수 / minsu@gmail.com
지훈 / jihoon@gmail.com
현우 / hyunwoo@gmail.com
```

초대받은 사용자는 초대 링크를 통해 여행방에 접근할 수 있다. 단, 실제 입장 권한은 초대 링크가 아니라 Google 로그인 계정의 이메일을 기준으로 판단한다.

### 8.3 초대 링크의 역할

초대 링크는 여행방 접근을 위한 안내 경로일 뿐, 권한 자체가 아니다.

초대 링크가 외부에 공유되더라도 로그인한 Google 계정이 초대 이메일과 일치하지 않으면 여행방에 접근할 수 없다.

### 8.4 초대 플로우

1. 어드민이 Google 로그인
2. 여행방 생성
3. 친구 이름과 이메일 입력
4. 멤버 초대 생성
5. 친구에게 초대 링크 공유
6. 친구가 초대 링크 접속
7. Google 로그인
8. 로그인 이메일과 초대 이메일 비교
9. 일치하면 참여 가능
10. 불일치하면 접근 차단

### 8.5 초대 이메일 정규화

초대 이메일과 로그인 이메일을 비교할 때는 정규화한 값을 사용한다.

- 공통: 앞뒤 공백 제거, 소문자화
- @gmail.com / @googlemail.com: 로컬부의 점(.) 제거, '+' 이후 문자열 제거 (예: Min.Su+trip@Gmail.com → minsu@gmail.com)

정규화한 이메일은 trip_members.normalized_invited_email에 저장하고, 멤버 매칭과 중복 초대 검사에 사용한다. 원본 입력 이메일도 함께 보관한다.

### 9.3 Member 권한

일반 멤버는 다음 작업을 수행할 수 있다.

- 여행방 조회
- 멤버 목록 조회
- 지출 추가
- 본인이 작성했거나 본인이 결제자인 지출 수정
- 본인이 작성했거나 본인이 결제자인 지출 삭제
- 전체 지출 목록 조회
- 정산 결과 조회
- 본인 표시 이름 수정
- AI 지출 입력 사용

### 9.4 권한 제한

일반 멤버는 다음 작업을 수행할 수 없다.

- 여행방 삭제
- 여행방 설정 수정
- 멤버 초대
- 다른 멤버 비활성화
- 본인이 작성하지도 결제하지도 않은 지출 삭제
- 초대 이메일 변경
- 정산 확정
- 정산 잠금 해제

### 9.5 어드민 승계 정책

- 여행방에는 항상 active 어드민이 1명 존재해야 한다. (불변식)
- 어드민은 참여 완료 상태의 다른 멤버에게 어드민 권한을 양도할 수 있다. 양도하면 기존 어드민은 일반 멤버가 된다.
- 마지막 어드민은 양도를 완료하기 전에는 탈퇴하거나 자신을 비활성화할 수 없다.
- 어드민이 양도 없이 사라진 비정상 상황(계정 삭제, 장기 미접속 등)에서는 서비스 운영자가 본인 확인 절차를 거쳐 어드민을 이전한다. 이는 MVP에서 자동화하지 않고 운영 프로세스로 처리한다.

## 10. 멤버 상태 정책

멤버는 다음 상태를 가진다.

| 상태 | 설명 |
|---|---|
| 초대됨 | 어드민이 이메일을 등록했지만 아직 참여하지 않은 상태 |
| 참여 완료 | Google 로그인 후 여행방 참여가 완료된 상태 |
| 비활성화 | 어드민이 접근을 제한한 상태 |
| 초대 만료 | 초대가 취소되었거나 유효기간이 지난 상태 |

### 10.1 초대됨

어드민이 친구의 이메일을 등록했지만 친구가 아직 Google 로그인 후 참여하지 않은 상태다.

### 10.2 참여 완료

친구가 초대 링크를 통해 접속하고, Google 로그인 이메일이 초대 이메일과 일치하여 여행방 참여가 완료된 상태다.

### 10.3 비활성화

어드민이 해당 멤버의 접근을 막은 상태다.

비활성화된 멤버는 여행방에 접근할 수 없다. 단, 기존 지출 데이터의 정산 무결성을 위해 비활성화된 멤버의 과거 지출 기록은 삭제하지 않는다.

비활성화 멤버는 **접근은 불가하나 정산 계산·정산표 표시 대상에는 남는다.** 정산표·멤버 목록·송금 항목에는 `display_name`(없으면 fallback: `display_name → user.name → 마스킹된 초대 이메일 → "탈퇴한 멤버"`)으로 '(비활성)' 상태 라벨과 함께 표시한다. 접근 차단과 정산 표시는 분리한다.

### 10.4 초대 만료

초대 링크가 만료되었거나 어드민이 초대를 취소한 상태다.

초대 토큰은 무작위로 생성하고 DB에는 해시(invite_token_hash)로 저장하며 만료 시점(invite_token_expires_at)을 둔다. 초대 재발송 시 이전 토큰을 폐기하고 새로 발급하며, 초대 취소 시 토큰을 무효화한다. 토큰은 접근 안내용 포인터일 뿐 권한이 아니다(8.3).

### 10.5 탈퇴 및 초대 거절

- 멤버는 본인 의사로 여행방에서 나갈 수 있다(탈퇴). 단, 마지막 어드민은 양도 전에는 탈퇴할 수 없다(9.5).
- 탈퇴해도 정산 무결성을 위해 과거 지출 기록은 삭제하지 않는다(10.3과 동일 원칙).
- 초대받은 사용자는 초대를 거절할 수 있으며, 거절 시 상태는 초대 만료로 처리한다.

## 11. DB 기반 권한 관리

### 11.1 기본 원칙

모든 여행방 데이터 접근은 DB의 멤버십 정보를 기준으로 제어한다.

사용자가 특정 여행방에 접근하려면 다음 조건을 만족해야 한다.

```txt
현재 로그인한 사용자가 해당 여행방의 참여 완료 상태 멤버이거나 어드민이어야 한다.
```

식별자 체계는 다음과 같이 분리한다.

- 전역 사용자 식별자: users.google_account_id (안정 식별자). email은 변동 속성이다.
- 여행방 내 정체성: trip_members.id. 지출·정산·권한 판정의 1차 식별자다.
- expenses의 결제자·참여자·작성자는 모두 trip_members.id를 참조한다.
- 초대만 된 멤버(user_id = null)도 trip_members 행으로 존재하므로, 참여 전에도 결제자·참여자로 지정할 수 있다. (총무 대리입력 지원)

## 12. 결제 방식 정책

### 12.1 결제 방식 구분

지출은 결제 방식에 따라 구분할 수 있다.

기본 결제 방식은 다음과 같다.

- 현금
- 카드
- 교통카드
- 간편결제
- 기타

### 12.2 결제 방식이 필요한 이유

현금 결제는 보통 현지 통화 기준 정산이 자연스럽다. 반면 카드 결제는 실제 정산 통화 청구 금액이 존재할 수 있기 때문에 카드 청구 금액 기준 정산이 더 정확할 수 있다.

예시:

```txt
현금 결제:
1,000 THB × 지출 시점 기준 환율

카드 결제:
실제 카드 청구 금액 39,800 KRW
```

## 13. 금액 저장 정책

### 13.1 기본 원칙

모든 지출은 현지 통화 금액과 정산 통화 금액을 모두 저장한다.

지출 입력 시 기본 통화는 여행방의 기본 현지 통화이며, 시스템은 지출 사용 일자 기준 환율을 적용하여 정산 통화 금액을 함께 계산한다.

### 13.2 저장하는 금액 정보

각 지출은 다음 금액 정보를 가진다.

- 현지 통화 금액
- 현지 통화 코드
- 정산 통화 금액
- 정산 통화 코드
- 적용 환율
- 환율 기준 일자
- 정산 통화 금액 출처
- 결제 방식
- 정산 포함 여부

모든 금액은 해당 통화의 최소단위 정수로 저장한다. 통화별 최소단위는 18.2.1의 정의를 따른다. (예: KRW·JPY·VND = 1, USD·EUR·THB = 1/100)

### 13.3 예시

```txt
지출명: 공항 택시
현지 금액: 1,000 THB
정산 통화: KRW
적용 환율: 1 THB = 37.9 KRW
정산 금액: 37,900 KRW
환율 기준 일자: 2026.08.04
결제 방식: 현금
```

## 14. 환율 적용 정책

### 14.1 기본 원칙

정산 통화 금액은 여행방 전체 고정 환율이 아니라 지출 일자 기준 환율을 적용한다.

즉, 같은 여행 안에서도 지출 일자에 따라 적용 환율이 달라질 수 있다.

### 14.2 환율 기준 일자

환율 기준 시점은 지출의 사용 일자(날짜 단위)를 따른다. 지출에 시각 정보가 있어도 해당 일자의 대표 환율을 적용한다. **사용 일자는 여행방 현지 시간대 기준**이다 — `exchange_rate_date = (spent_at AT TIME ZONE trips.timezone)::date`. (자정 근처 지출의 날짜 어긋남 방지)

예시:

```txt
8월 3일 오전 식비 → 8월 3일 기준 환율 적용
8월 5일 저녁 쇼핑 → 8월 5일 기준 환율 적용
```

### 14.3 적용 환율 우선순위

환산("converted")으로 정산 통화 금액을 계산할 때, 적용 환율은 다음 우선순위로 결정한다. (정산 통화 금액 자체의 출처 우선순위는 18.5)

1. 사용자가 직접 입력/수정한 환율 (manual)
2. 지출 일자 기준 자동 조회 환율 (auto)
3. 마지막으로 성공적으로 가져온 환율 캐시 (last_known)
4. 여행방 기본 환율 (trip_default)

각 지출에는 적용된 환율 출처(exchange_rate_source)를 저장한다. auto가 아닌 값(manual/last_known/trip_default)이 적용된 경우, 사용자에게 자동 환율이 아닌 대체 환율로 계산되었음을 표시한다.

예시:

```txt
환율 정보를 불러오지 못해 마지막으로 확인된 환율을 적용했어요.
```

적용 환율은 지출 저장 시점에 exchange_rate로 동결하며, 이후 자동 재조회로 덮어쓰지 않는다. 환율은 금액이 아니므로 18.2.1의 정수 규칙을 적용하지 않고, 충분한 소수 정밀도(소수점 이하 6자리 권장)로 저장한다.

## 18. 정산 기준 및 계산 정책

### 18.1 기본 원칙

최종 정산은 현지 통화 기준 정산과 정산 통화 기준 정산을 분리해서 보여준다.

- 현지 통화 기준 정산은 참고용 또는 현지 현금 정산용으로 제공한다.
- 정산 통화 기준 정산은 실제 송금용으로 기본 제공한다.
- 기본 화면에서는 정산 통화 기준 정산을 강조하고, 현지 통화 기준 정산은 접기/탭으로 제공해 인지 부하를 줄인다.

### 18.2 멤버별 부담액 계산

각 지출마다 참여자들이 동일하게 나누어 부담하되, 나누어떨어지지 않을 때의 잔여 처리까지 결정적으로 정의한다.

#### 18.2.1 연산 원칙

- 모든 금액은 통화 최소단위 정수로 저장·계산한다. (KRW·JPY·VND 등 = 1, USD·EUR·THB 등 = 1/100) 통화별 최소단위는 ISO 4217 exponent를 따르며, 부록(48장) 표로 관리한다.
- 정산 계산에는 부동소수점을 사용하지 않는다.
- 모든 지출의 참여자는 최소 1명 이상이어야 한다.

#### 18.2.2 지출별 부담액 분배

각 지출의 부담액은 지출 단위에서 정수로 확정하여 저장한다.

```txt
1. 기본 부담액 = floor(지출 금액 / 참여자 수)        // 통화 최소단위 정수
2. 잔여 = 지출 금액 - (기본 부담액 × 참여자 수)       // 0 ≤ 잔여 < 참여자 수
3. 잔여는 참여자를 member_id 오름차순으로 정렬해 앞에서부터 1최소단위씩 배분한다.
4. 불변식: Σ(참여자 부담액) == 지출 금액
   위반 시 계산 실패로 처리하고 저장을 차단한다.
```

예시 (10,000 KRW, 참여자 3명):

```txt
기본 부담액 = floor(10,000 / 3) = 3,333
잔여 = 10,000 - 9,999 = 1
member_id 오름차순 첫 1명에게 +1
→ 3,334 / 3,333 / 3,333  (합계 10,000)
```

#### 18.2.3 멤버별 집계

각 멤버별로 다음 값을 계산한다.

```txt
총 결제 금액 = Σ(내가 결제자인 정산 대상 지출 금액)
총 부담 금액 = Σ(내가 참여자인 지출에서 확정된 내 부담액)
순정산 금액 = 총 결제 금액 - 총 부담 금액
불변식: Σ(전 멤버 순정산 금액) == 0
```

#### 18.2.4 이중 통화축

현지 통화 정산과 정산 통화 정산은 각각 독립적으로 위 분배·집계를 수행한다.

- 현지 통화 정산: 각 지출의 현지 통화 금액을 기준으로 분배한다.
- 정산 통화 정산: 각 지출의 정산 통화 금액(13.2에 정수로 확정 저장된 settlement_amount, 즉 카드 청구 금액 또는 환산 금액)을 기준으로 분배한다.

두 통화축 모두 18.2.2의 불변식을 독립적으로 만족한다.

### 18.3 해석

- 순정산 금액이 양수이면 받아야 할 돈이 있다.
- 순정산 금액이 음수이면 보내야 할 돈이 있다.
- 순정산 금액이 0이면 정산 완료 상태다.

### 18.4 최소 송금 로직

입력은 멤버별 순정산 금액(통화 최소단위 정수, 합계 0 보장)이다.

```txt
1. 순정산 > 0인 받을 사람 목록과 순정산 < 0인 보낼 사람 목록을 만든다.
2. 두 목록을 각각 금액 내림차순으로, 금액이 같으면 member_id 오름차순으로 정렬한다. (결정성 보장)
3. 가장 많이 받을 사람과 가장 많이 보낼 사람을 매칭한다.
4. 둘 중 작은 금액만큼 송금 항목을 만들고, 양쪽 잔액에서 차감한다. 잔액이 0이 된 사람은 목록에서 제거한다.
5. 모든 잔액이 정확히 0이 될 때까지 반복한다.
```

- 정수 연산이므로 매 단계 최소 1명의 잔액이 0이 되어, 송금 항목은 (멤버 수 - 1)개 이내에서 종료된다. (무한 루프 불가)
- 정렬 기준이 결정적이므로 동일한 입력은 항상 동일한 송금 리스트를 산출한다. (31장 정산 확정 스냅샷 재현의 전제)
- 현지 통화 정산표와 정산 통화 정산표는 각각 독립적으로 이 로직을 실행하므로 송금 그래프가 서로 다를 수 있다. 이는 지출별 환율 차이에 따른 정상 결과이며, 실제 송금 기준은 정산 통화 표 하나다. (38.11)

### 18.5 정산 통화 금액 출처 우선순위

정산 통화 금액(settlement_amount)의 출처는 다음 우선순위를 따르며, 각 지출에 settlement_amount_source를 저장한다.

1. 카드 청구 정산 통화 금액 — 사용자가 직접 입력했거나, AI가 추출하고 사용자가 확인한 실제 청구 금액 (card_billed)
2. 현지 통화 금액 × 적용 환율로 계산한 환산 금액 (converted)

"converted"일 때 적용 환율의 우선순위는 14.3을 따른다.

- card_billed인 경우 exchange_rate는 환산에 사용되지 않으므로 **선택적이며 null일 수 있다.** 참고용 내재 환율(settlement_amount/local_amount)은 표시용으로 계산할 수 있으나 정산 계산에는 영향을 주지 않는다. exchange_rate_date는 두 경우 모두 지출 일자 기준 대표 환율의 날짜를 가리킨다(14.2).

### 18.6 현지 통화 정산 금액

현지 통화 정산 금액은 사용자가 입력하거나 AI가 추출한 현지 통화 금액을 기준으로 한다.

다중 통화 여행에서는 현지 통화 정산을 통화별로 분리해서 제공한다.

현지 통화 기준 정산표는 현지 현금 정산 시 참고용이며, 지출별 환율 차이로 인해 정산 통화 표와 송금 방향·필요 여부가 다를 수 있다. 두 표는 합산하거나 교차 정산하지 않으며, 실제 송금 기준은 정산 통화 표 하나다.

## 21. 정산 제외 및 개인 기록용 지출

### 21.1 기능 개요

사용자는 특정 지출을 정산에서 제외하고 기록용으로만 저장할 수 있다.

예시:

- 개인 쇼핑
- 혼자 산 기념품
- 개인 교통비
- 개인 간식
- 정산에 포함하지 않을 개인 결제

### 21.2 지출 상태

지출은 다음 상태 중 하나를 가질 수 있다.

| 상태 | 설명 |
|---|---|
| 정산 포함 | 최종 정산에 포함되는 지출 |
| 개인 지출 | 개인 기록용이며 정산에서 제외되는 지출 |
| 기록만 저장 | 여행 기록에는 남기지만 정산에는 포함하지 않는 지출 |

이 3개 상태는 `expenses.expense_settlement_state`(`included | personal | record_only`) 단일 enum으로 저장하며 DB CHECK 제약으로 도메인을 강제한다(두 boolean 조합이 만드는 모순 상태 방지). 지출 추가 화면의 정산 포함 토글은 이 enum(included vs 그 외)에서 파생되며, personal과 record_only는 둘 다 정산에서 제외되지만 저장 시 구분된다.

### 21.3 정책

- 정산 제외 지출은 총 여행 지출 통계에는 포함할 수 있다.
- 정산 제외 지출은 최종 정산 계산에는 포함하지 않는다.
- 지출 목록에서는 정산 포함/제외 여부를 명확히 표시한다.

## 22. 스마트 지출 입력

### 22.1 기능 개요

스마트 지출 입력은 사용자가 지출 폼을 직접 모두 작성하지 않아도 자연어 문장, 카드 승인 문자, 카드 앱 결제 내역 텍스트, 영수증 사진 등을 통해 지출 정보를 입력할 수 있게 하는 기능이다.

시스템은 입력 내용을 분석하여 다음 정보를 자동으로 추출한다.

- 지출명
- 결제 금액
- 현지 통화
- 정산 통화 청구 금액
- 결제자
- 참여자
- 카테고리
- 결제일
- 결제 방식
- 메모

### 22.2 저장 전 확인 원칙

AI가 분석한 지출은 자동으로 저장하지 않는다.

반드시 다음 과정을 거친다.

```txt
입력
→ AI 분석
→ 지출 후보 생성
→ 사용자 확인/수정
→ 저장
```

정산 앱은 돈과 관련된 데이터이므로 AI 분석 결과는 항상 사용자의 최종 확인을 받아야 한다.

- 금액·통화 등 금전에 직접 영향을 주는 필드는 확인 화면에서 강조 표시하고, 사용자가 명시적으로 확인해야 card_billed 또는 현지 금액으로 확정한다.
- AI가 추출한 결제자·참여자 이름이 멤버와 매핑되지 않으면 "미지정"으로 두고 사용자가 확인 화면에서 멤버를 선택한다. 자동 추정 매핑은 신뢰도가 높을 때만 제안하되 항상 사용자 확인을 거친다.
- 각 지출에는 입력 출처(input_source: 직접입력 / AI 한 줄 / 카드 문자 / 영수증 / 카드 캡처)를 기록한다.

### 22.3 다국가 AI 입력 고려사항

AI는 국가와 통화에 따라 다른 영수증 형식, 숫자 표기, 통화 기호를 처리해야 한다.

예시:

| 국가/지역 | 통화 | 예시 표기 |
|---|---|---|
| 일본 | JPY | ¥1,280 |
| 대만 | TWD | NT$120 |
| 태국 | THB | ฿250 |
| 미국 | USD | $18.50 |
| 유럽 | EUR | €24.30 |
| 영국 | GBP | £15.20 |
| 베트남 | VND | 120,000₫ |

AI가 인식한 통화가 여행방 기본 현지 통화와 다를 경우 사용자에게 확인을 요청한다.

### 22.4 AI 입력 안전·비용 정책

- 외부 LLM·OCR 제공자에는 정산에 필요한 정보 추출만을 목적으로 데이터를 전송하며, 학습 미사용·미보존을 보장하는 제공자(DPA 체결)만 사용한다.
- 카드 문자 등 텍스트 입력은 전송 전 클라이언트에서 카드번호·승인번호를 마스킹한다.
- 영수증·카드 캡처 이미지는 압축(26.1) 후 자체 스토리지에 보존한다. 외부 AI 제공자에는 학습 미사용·미보존으로 전송하되, 자체 보존 이미지는 접근통제·보존기간·삭제 규칙(43장)을 따른다.
- AI 입력(Phase 2 도입)은 초기 무료로 제공하되, 사용자·여행방당 호출 레이트리밋과 환율 캐싱으로 비용과 남용을 통제한다. 비즈니스 모델은 추후 결정한다.
- AI 분석은 비대화형(one-shot) 호출로 구현한다. 텍스트 기반 입력(자연어 한 줄, 카드 문자)은 Codex 기반 비대화형 호출을 사용하며, 대화형 챗봇 형태는 제공하지 않는다. (1.1, 5.3과 일치)
- 영수증·카드 캡처 등 이미지 기반 입력은 이미지 인식(비전·OCR)이 필요하므로 텍스트 입력과 다른 경로로 처리하며, 구체 방식은 해당 기능이 속한 Phase 3에서 확정한다.

## 26. 이미지 업로드 정책

### 26.1 업로드 전 이미지 처리

모바일 네트워크 환경을 고려하여 이미지는 업로드 전 압축할 수 있다.

권장 정책:

- 긴 변 기준 1600px 이하로 리사이즈
- 이미지 품질 70~80% 수준으로 압축
- 원본 이미지는 저장하지 않고 분석용 이미지 또는 압축 이미지 사용

### 26.2 업로드 실패 처리

여행 중에는 네트워크가 불안정할 수 있으므로 업로드 실패에 대비해야 한다.

업로드 실패 시 제공해야 할 옵션:

- 다시 시도
- 이미지 다시 선택
- 직접 입력으로 전환
- AI 한 줄 입력으로 전환

### 26.3 민감정보 처리

영수증 또는 카드 앱 캡처에는 민감한 정보가 포함될 수 있다.

서비스는 다음 정보를 저장하지 않는 것을 원칙으로 한다.

- 카드번호
- 승인번호
- 계좌번호
- 보안코드
- 불필요한 개인 식별 정보

서비스는 정산에 필요한 정보만 저장한다.

또한 위 민감정보는 구조화 데이터로 저장하지 않으며 외부로의 전송도 하지 않는 것을 원칙으로 한다. 텍스트 입력은 전송 전 클라이언트에서 마스킹하고, 이미지 입력은 학습 미사용·미보존을 보장하는 제공자로만 전송한다. (22.4)

단, 보존하는 영수증·카드 이미지에는 민감정보가 시각적으로 포함될 수 있으므로, 인증된 여행방 멤버만 접근(공개 URL 금지)·보존기간 제한·사용자 삭제·저장소 암호화(at-rest)로 보호한다. (43장)

저장 대상 정보:

- 결제처
- 결제일
- 결제 금액
- 통화
- 결제 방식
- 카테고리
- 메모

### 28.1 기능 개요

카드 문자, 영수증 사진, 직접 입력, AI 입력이 섞이면 같은 지출이 중복 등록될 가능성이 있다.

서비스는 새 지출을 저장하기 전에 기존 지출과 비교하여 중복 가능성을 탐지한다.

### 28.2 탐지 기준

- 결제일
- 결제 시간
- 결제처
- 금액
- 통화
- 결제자
- 카테고리

### 28.3 UX 예시

```txt
비슷한 지출이 이미 있어요.

기존 지출:
7-ELEVEN / 120 TWD / 8월 4일 19:32 / 상욱 결제

새로 추가하려는 지출:
```

### 28.4 정책

- 직접 입력에도 중복 감지를 적용할 수 있다.
- 중복 감지는 자동 삭제가 아니라 저장 전 경고로만 동작한다.

## 29. 정산 전 점검 체크리스트

### 29.1 기능 개요

정산 버튼을 누르면 바로 결과만 보여주는 것이 아니라, 먼저 빠진 지출이나 이상한 지출이 없는지 점검한다.

### 29.2 점검 항목

정산 전 점검은 다음 항목을 확인한다.

- 숙소비 입력 여부
- 날짜별 식비 기록 누락 가능성
- 날짜별 교통비 기록 누락 가능성
- 고액 지출 중 참여자가 1명뿐인 항목
- 전체 멤버 중 결제 기록이 전혀 없는 멤버
- 전체 멤버 중 참여 기록이 전혀 없는 멤버
- 금액이 비정상적으로 큰 지출
- 카드/영수증 기반 지출 중 참여자가 선택되지 않은 항목
- 정산 제외 지출이 너무 많은 경우
- 여행방 기본 통화와 다른 통화로 입력된 지출

### 29.3 UX 예시

```txt
정산 전 확인해볼 항목이 있어요.

1. 숙소비가 아직 입력되지 않았어요.
2. 8월 4일 저녁 식비 기록이 없어요.
3. 민수는 지출 참여 기록은 있지만 결제 기록이 없어요.
4. 12,800엔 이상 지출 중 참여자가 1명뿐인 항목이 있어요.
5. 여행 기본 통화와 다른 통화로 입력된 지출이 2개 있어요.

그래도 정산을 진행할까요?
```

### 29.4 정책

- 정산 전 점검은 정산을 막는 기능이 아니라 확인을 돕는 기능이다.
- 사용자는 경고를 확인한 뒤 정산을 계속 진행할 수 있다.
- 초기에는 규칙 기반으로 제공하고, 이후 AI 기반 누락 탐지로 확장할 수 있다.

## 30. 지출 수정 히스토리

### 30.1 기능 개요

정산 앱은 신뢰가 중요하므로 지출이 수정되었을 때 누가 언제 무엇을 변경했는지 기록한다.

### 30.2 MVP 정책

MVP에서는 최소한 다음 정보를 저장한다.

- 마지막 수정 시간

## 31. 정산 확정 및 잠금

### 31.1 기능 개요

여행이 끝나고 정산 결과를 공유한 뒤 누군가 지출을 수정하면 결과가 바뀔 수 있다.

이를 방지하기 위해 어드민은 최종 정산을 확정하고 여행방을 잠글 수 있다.

### 31.2 정산 확정 플로우

```txt
정산 화면
→ 정산 전 점검
→ 정산 결과 확인
→ 정산 확정 (결과 스냅샷 생성)
→ 지출 추가/수정/삭제 제한
```

### 31.3 확정 전 안내

```txt
정산을 확정하면 지출 추가, 수정, 삭제가 제한됩니다.
그래도 확정할까요?
```

### 31.4 확정 후 안내

```txt
이 여행은 정산 확정 상태입니다.
지출을 수정하려면 어드민이 정산 잠금을 해제해야 합니다.
```

### 31.5 정책

- 정산 확정은 어드민만 할 수 있다.
- 정산 확정 후 일반 멤버는 지출을 추가/수정/삭제할 수 없다.
- 정산 확정 후에도 지출 목록과 정산 결과는 조회할 수 있다.
- 어드민은 필요한 경우 정산 잠금을 해제할 수 있다.
- 잠금 해제 후 지출이 변경되면 기존 공유 정산 결과와 달라질 수 있음을 표시한다.
- 정산 확정 시 그 시점의 정산 결과(송금 항목·총지출·멤버별 요약)를 settlements 스냅샷으로 동결한다.
- 잠금 해제 후 재확정하면 새 version 스냅샷을 만들고, 직전 active 스냅샷과의 차이(변경된 송금 항목)를 표시한다.

### 31.6 정산 상태와 스냅샷

여행방은 settlement_status로 정산 상태를 가진다.

| 상태 | 설명 |
|---|---|
| open | 정산 전. 지출 추가/수정/삭제 가능 |
| finalized | 정산 확정·잠금. 지출 변경 제한, active 스냅샷 존재 |

- open → finalized: 어드민이 확정한다. 현재 지출 집합을 단일 트랜잭션에서 스냅샷으로 동결한다.
- finalized → open: 어드민이 잠금을 해제한다. **직전 active 스냅샷은 그대로 유지되어 재확정 시 diff 기준이 된다(잠금 해제만으로 superseded로 바꾸지 않는다).** 재확정이 성공하면 그때 직전 active를 superseded로 전환한다.
- 동시 수정 충돌을 막기 위해 지출은 version(낙관적 잠금)을 가지며, 확정 트랜잭션 중 변경된 지출이 감지되면 확정을 중단하고 재확인을 요청한다.

### 31.7 송금 완료 추적

정산 확정 후, 각 송금 항목(settlement_transfers)에 대해 실제 송금 완료 여부를 수동으로 추적한다. (은행 송금 연동은 하지 않는다. 5.3)

- 송금 완료 표시는 수취인(받는 사람) 본인 또는 어드민이 할 수 있다.
- 표시 시 payment_status를 paid로 바꾸고 paid_at과 marked_by_member_id를 기록한다.
- 정산 화면은 송금 항목별로 대기/완료 상태와 전체 정산 완료율을 보여준다.
- 송금 추적은 확정 스냅샷의 송금 항목에 대해서만 동작한다. 잠금 해제 후 재확정으로 새 스냅샷이 생기면 송금 완료 상태는 새 스냅샷 기준으로 다시 확인한다.

## 34. 비기능 요구사항

### 34.1 사용성

- 모바일 화면에서 한 손으로 지출을 입력할 수 있어야 한다.
- 지출 직접 입력은 빠르게 완료할 수 있어야 한다.
- Google 로그인 이후 초대받은 여행방에 자연스럽게 진입해야 한다.
- 초대받지 않은 계정은 왜 접근할 수 없는지 명확히 안내해야 한다.
- 금액은 현지 통화와 정산 통화를 함께 보여준다.
- AI 분석 결과는 사용자가 쉽게 수정할 수 있어야 한다.
- 국가/통화가 달라도 동일한 흐름으로 입력할 수 있어야 한다.

### 34.2 성능

- 여행방 홈 첫 로딩은 2초 이내를 목표로 한다.
- 지출 저장 후 목록 반영은 1초 이내를 목표로 한다.
- 정산 계산은 즉시 처리 가능해야 한다.
- MVP 기준 한 여행방당 지출 300개까지 무리 없이 처리해야 한다.

### 34.3 접근성

- 버튼 터치 영역은 최소 44px 이상으로 한다.
- 금액, 정산 결과는 색상만으로 의미를 전달하지 않는다.
- 받을 돈/보낼 돈은 텍스트 라벨을 함께 표시한다.
- 다크모드 대응은 MVP 이후로 미룬다.
- WCAG 2.1 AA를 목표로 한다: 모든 이미지·아이콘 대체텍스트, 폼-라벨 연결, 정산표 시맨틱 마크업, 키보드 전체 조작, 동적 결과 변경 aria-live, 최소 명도 대비 4.5:1.

### 34.4 보안

- Google 로그인 기반으로 사용자를 식별한다.
- 초대된 Google 이메일과 로그인 이메일이 일치해야 여행방에 접근할 수 있다.
- 초대 링크 자체는 권한으로 사용하지 않는다.
- 여행방 ID는 예측하기 어려운 UUID를 사용한다.
- 민감한 개인정보는 최소한으로 수집한다.
- 카드번호, 승인번호, 계좌번호 등 민감정보는 저장하지 않는다.
- 비활성화된 멤버는 여행방에 접근할 수 없다.
- Google `email_verified=true`로 검증된 이메일만 초대 매칭에 사용한다.
- 세션 쿠키는 `HttpOnly`·`Secure`·`SameSite`로 발급하고, FE/BE가 서로 다른 서브도메인이면 쿠키 `Domain`과 CORS allowed origin(+`Allow-Credentials`)을 명시 allowlist로 제한한다(와일드카드 금지). 상태변경 요청에는 CSRF 방어를 둔다. (구현 정책은 tech-stack §5.2·architecture §4.7)
- 초대 링크 토큰은 무작위로 생성하고 DB에는 해시로 저장하며, 만료일을 두고 재발송·취소 시 이전 토큰을 폐기한다. 토큰은 접근 안내용일 뿐 권한이 아니다(8.3).

### 34.5 오류·빈 상태·로딩 처리 표준

화면별로 흩어지지 않도록 전역 표준을 둔다.

- 오류 유형별 표준 메시지·동작: 네트워크 실패(재시도), 권한 거부(접근 불가 사유 안내), 입력 검증 오류(필드 강조), 서버 오류(재시도·문의).
- 모든 주요 화면(32장)은 빈 상태·로딩 상태·오류 상태를 정의한다.
- 정산처럼 부분 실패가 가능한 계산은 어떤 항목이 왜 계산되지 않았는지 표시한다.
- 만료된 초대·차단된 접근은 사용자에게 이유를 명확히 안내한다.

## 35. 성공 지표

### 35.1 MVP 성공 기준

- 사용자가 Google 로그인으로 여행방을 생성할 수 있다.
- 어드민이 여행 국가와 기본 현지 통화를 설정할 수 있다.
- 어드민이 친구 이메일을 입력해 초대할 수 있다.

- 지출을 현지 통화/정산 통화 기준으로 저장할 수 있다.
- 지출 시점 기준 환율이 적용된다.
- 정산 제외 지출이 최종 정산에서 제외된다.
- 현지 통화 기준 정산과 정산 통화 기준 정산이 분리되어 표시된다.
- 정산 결과를 복사할 수 있다.

### 35.2 정량 지표

- 여행방 생성 완료율
- 초대 링크 참여 완료율
- 초대 이메일 불일치 차단 횟수
- 지출 추가 완료율
- 여행방당 평균 지출 개수
- 정산 결과 복사 횟수
- 정산 확정 사용률
- AI 입력 사용률
- 영수증/카드 이미지 분석 성공률
- 국가별 여행방 생성 비율
- 통화별 지출 입력 비율

### 35.3 정성 지표

- 친구들이 설명 없이 사용할 수 있는가
- 여행 중 입력이 귀찮지 않은가
- 최종 정산 결과를 신뢰할 수 있는가
- 현지 통화/정산 통화 정산 분리가 이해하기 쉬운가
- AI 분석 결과를 수정하기 쉬운가
- 기존 메모장/계산기 방식보다 편한가
- 일본 외 국가에서도 자연스럽게 사용할 수 있는가

### 35.4 수용 기준 (Acceptance Criteria)

기능 존재 여부를 넘어 검증 가능한 기준을 둔다.

- 정산 합계 불일치 허용 = 0 (18.2.1 불변식).
- 환율 적용 시점·출처가 결정적이고 재현 가능 (14.3).
- 성능 목표는 측정 조건과 함께 정의 (기기·네트워크·지출 300건 기준, 34.2).
- AI/OCR 목표 인식률과 초대 이메일 정규화 일치 케이스 표를 검증 가능한 형태로 둔다.

## 38. 핵심 UX 원칙

1. 여행방은 어드민이 초대한 사람만 접근할 수 있어야 한다.
2. 초대 링크가 유출되어도 초대받은 Google 계정이 아니면 접근할 수 없어야 한다.
3. 일본뿐 아니라 다른 국가 여행에서도 자연스럽게 사용할 수 있어야 한다.
4. 지출 입력은 빠르게 할 수 있어야 한다.
5. 돈과 관련된 AI 분석 결과는 자동 저장하지 않는다.
6. AI는 사용자의 입력 부담을 줄이는 보조 수단이어야 한다.
7. 모든 지출은 현지 통화와 정산 통화를 함께 저장한다.
8. 정산 통화 금액은 지출 사용 일자 기준 환율을 적용한다.
9. 카드 청구 정산 통화 금액이 있는 경우 해당 금액을 우선 반영할 수 있다.
10. 최종 정산은 현지 통화 기준과 정산 통화 기준을 분리해서 보여준다.
11. 실제 송금에는 정산 통화 기준 정산을 기본으로 사용한다.
12. 현지 통화 기준 정산은 참고용 또는 현지 현금 정산용으로 제공한다.
13. 정산 제외 지출은 명확히 표시한다.
14. 영수증/카드 캡처는 정산에 필요한 정보만 추출해야 한다.
15. 민감정보는 저장하지 않는다.
16. 최종 정산 결과는 신뢰 가능하고 공유하기 쉬워야 한다.
17. 정산 확정 후 지출 변경을 제한해 결과 변경을 방지한다.
18. 모바일 웹에서 촬영, 업로드, 확인, 저장까지 자연스럽게 이어져야 한다.

## 39. 오픈 이슈

1. (해결) 환율 데이터 소스로 **Open Exchange Rates**(일 단위·historical·9통화 커버)를 선정, secondary로 currencyapi.com을 둔다. last_known 캐시와 여행방 기본 환율 fallback(14.3)으로 장애에 대비한다. (조사·파이프라인: tech-stack §7·§9, `docs/plans/2026-06-25-fx-pipeline-design.md`)
2. (해결) 환율은 날짜 단위로 적용한다. 지출에 시각 정보가 있어도 해당 일자의 대표 환율을 사용한다. (14.1~14.3)
3. 정산 통화 청구 금액과 자동 환산 금액이 다를 때 UI에서 어떻게 설명할 것인가?
4. (해결) 정산 제외 지출은 총 여행 지출 통계에는 포함하고, 최종 정산 계산에서는 제외한다. (21.3)
5. (해결) 초대 이메일 수정은 참여(join) 전까지만 유효하다. 참여 후에는 trip_members.user_id로 멤버십이 고정되므로 초대 이메일 변경은 권한에 영향을 주지 않는다. (7.3, 11.1)
6. (해결) 비활성화·탈퇴(익명화 포함) 멤버라도 정산표·멤버별 요약·송금 항목(settlement_transfers의 from/to)에는 정산 무결성을 위해 계속 표시된다. 접근 차단(권한 박탈)과 정산 표시(과거 집계 대상 유지)를 분리하며, 표시명은 display_name fallback 체인을 따른다. (10.3, 43.2, 31.7)
7. (변경) 영수증·카드 캡처 이미지는 자체 스토리지(홈랩 SSD)에 **보존**한다 — 접근통제·보존기간·삭제 §43, 압축 §26.1. 외부 AI 제공자엔 미보존 전송. 단일노드 SSD 유실 리스크는 백업으로 대비.
8. (해결) 정산 확정 시 결과를 settlements 스냅샷으로 동결하고, 잠금 해제 후 재확정 시 직전 active 스냅샷과 diff(변경된 송금 항목)를 표시한다. (31.5~31.7)
9. (보강) 중복 지출 감지는 규칙 기반 기본 민감도로 시작하고 저장 전 경고로만 동작한다(28.4). 사용자 민감도 조절은 MVP 이후로 둔다.
10. 정산 전 점검 체크리스트를 규칙 기반으로 시작할지, 초기부터 AI 기반으로 제공할지 결정해야 한다.
11. 다중 국가 여행에서 기본 현지 통화를 여러 개 허용할 시점을 언제로 할 것인가?
12. (해결) 통화별 소수점 처리 정책 — 18.2.1에서 통화 최소단위 정수 연산으로 확정. ISO 4217 exponent 기반 부록 표로 관리하며, 나머지는 18.2.2의 member_id 오름차순 배분 규칙을 따른다.
13. 현지 통화가 없는 카드 원화 결제만 있는 지출을 어떻게 표시할 것인가?

## 40. 최종 제품 방향 요약

본 서비스는 친구들과 해외여행 중 발생하는 지출을 기록하고 정산하는 모바일 웹 서비스다.

사용자는 Google 로그인을 통해 인증하며, 여행방은 어드민이 초대한 Google 계정만 접근할 수 있다. 어드민은 친구 이메일을 등록하여 멤버를 초대하고, 초대받은 사용자는 Google 로그인 후 여행방에 참여한다.

여행방은 여행 국가, 기본 현지 통화, 정산 통화를 가진다. 지출은 현지 통화 금액과 정산 통화 금액을 모두 저장한다. 정산 통화 금액은 지출 사용 일자 기준 환율을 적용하며, 카드 청구 정산 통화 금액이 있는 경우 해당 금액을 우선 반영할 수 있다.

최종 정산은 정산 통화 기준과 현지 통화 기준을 분리해서 보여준다. 실제 송금에는 정산 통화 기준 정산을 기본으로 사용하고, 현지 통화 기준 정산은 참고용 또는 현지 현금 정산용으로 제공한다.

AI 기능은 대화형 챗봇이 아니라 입력 보조 기능으로 제공된다. 사용자는 자연어 문장, 카드 승인 문자, 영수증 사진, 카드 앱 캡처 등을 통해 지출을 입력할 수 있으며, AI는 이를 분석해 지출 후보를 생성한다. 모든 AI 분석 결과는 사용자가 확인 및 수정한 뒤 저장된다.

정산 신뢰도를 높이기 위해 중복 지출 감지, 정산 전 점검 체크리스트, 지출 수정 히스토리, 정산 확정/잠금 기능을 제공한다.

## 41. 알림 정책

### 41.1 채널

알림은 이메일(트랜잭션)과 인앱 알림을 함께 사용한다.

- 이메일: 멤버 초대, 정산 확정, 공유 정산 변경 등 핵심 이벤트를 시스템이 자동 발송한다.
- 인앱: 앱 내 알림 목록과 읽음 표시·배지로 제공한다.
- 모바일 웹 환경을 고려해 웹푸시(PWA)는 MVP 이후로 둔다.

### 41.2 알림 이벤트

- 멤버 초대됨 (초대 대상에게 초대 이메일 자동 발송)
- 정산 확정됨
- 확정된 정산이 잠금 해제·재확정되어 결과가 바뀜
- (Phase 2) 능동 리마인더 (41.4)

### 41.3 초대 발송

어드민이 멤버를 초대하면 시스템이 초대 이메일을 자동 발송한다. 어드민은 초대 링크를 직접 복사해 공유할 수도 있다. (8.4)

### 41.4 능동 리마인더 (Phase 2)

입력 누락을 막기 위해 규칙 기반 리마인더를 제공한다.

- 여행 중 당일 지출 입력 리마인더
- 여행 종료일 전후 정산 점검 리마인더 (29장 정산 전 점검과 연계)

리마인더는 규칙 기반으로 시작하며 사용자가 끌 수 있다.

- 어드민이 입력하는 친구의 이름·이메일은 초대 목적에 한정해 처리한다.

### 42.5 데이터 처리 위치

- 데이터 처리·저장 리전과 외부 처리위탁(AI·OCR·환율 API) 현황을 처리방침에 명시한다. (22.4, 46장)

## 43. 데이터 보존·삭제·계정 탈퇴

### 43.1 삭제 방식

- 여행방·지출·이미지·감사로그의 삭제 방식(soft/hard)과 캐스케이드 규칙을 정의한다.
- 영수증·카드 캡처 이미지는 자체 스토리지에 보존하며, **인증된 여행방 멤버만 접근**(공개 URL 금지)한다. 지출/여행방 삭제 시 함께 파기(캐스케이드), 사용자는 개별 이미지를 삭제할 수 있다.

### 43.2 계정 탈퇴

- 사용자는 계정을 탈퇴할 수 있다.
- 탈퇴 시 본인의 PII(이메일·이름·프로필)는 삭제하거나 닉네임으로 치환(익명화)한다.
- 정산에 묶인 지출·부담 기록은 정산 무결성을 위해 익명화한 상태로 보존한다. (10.3, 10.5와 동일 원칙)
- 익명화된 탈퇴자도 정산표·송금 항목에 익명화 표시명(닉네임)으로 송금자/수취인으로 남을 수 있다. 송금 완료 표시(31.7)는 탈퇴자가 직접 할 수 없으므로 어드민이 대신 수행한다.

### 43.3 보존 기간

- 항목별 보존 기간과 파기 시점을 정의하고 처리방침에 고지한다. (42.2)
- 보존 이미지는 지출/여행방 수명에 종속(삭제 시 파기)하거나 별도 보존기간을 적용한다. 단일노드 SSD 저장이므로 유실 대비 백업(예: 주기적 외부 동기)과 저장소 암호화(at-rest)를 둔다.

## 44. 국제화(i18n)·표시 포맷

### 44.1 표시 포맷 표준

- 통화별 심볼·소수 자릿수·자리구분 규칙을 부록(48장) 표로 관리한다.
- 날짜·시간 표시 포맷을 1종으로 통일한다.
- 정산 통화 표기를 통일한다. (예: 'KRW'와 '원' 혼용 금지)

### 44.2 로케일 인지 숫자 파싱

- 입력·AI 파싱에서 자리구분/소수점 기호가 로케일에 따라 다름을 처리한다. (예: '1.000'을 천 단위로 볼지 소수로 볼지)

### 44.3 다국어

- MVP 기본 언어는 한국어이며, 다국어 UI는 Phase 이후 확장한다. 문자열은 i18n 대응이 가능하도록 분리한다.

## 45. 입력 검증·도메인 제약

- 여행 기간: 시작일 ≤ 종료일, 합리적 기간 길이 한도.
- 멤버 수: 상·하한 (최소 어드민 1명).
- 금액: 범위·소수 자릿수(통화 최소단위), 음수 정책(환불은 47장).
- 참여자: 지출당 최소 1명 (18.2.1).
- spent_at: 여행 기간을 벗어나면 확인 요청 (29장 점검과 연계).
- 검증은 입력 폼과 DB(제약·트랜잭션) 양쪽에 적용한다.

## 46. 운영·관측가능성

- 가용성 목표(업타임/SLA)를 정의한다.
- 데이터 백업 주기와 복구 목표(RPO/RTO)를 정의한다. (금전 데이터 유실은 핵심 가치 훼손)
- 애플리케이션·에러 로깅, 모니터링·알람을 둔다.
- 외부 API(환율·LLM·OCR) 의존성의 장애 격리·레이트리밋·비용 상한·캐싱을 둔다. (14.3, 22.4)
- 배포·롤백 전략을 정의한다.

## 47. 환불·취소 및 실거래 변형

### 47.1 환불·취소 (모델 준비)

- 환불·취소는 음수 금액 지출 또는 원지출 연결(refund_of_expense_id)로 모델링한다.
- 데이터 모델은 환불을 수용하도록 설계하되, 기능 제공 시점은 Phase 3로 둔다.
- 이미 공유·확정된 정산을 바꾸는 환불은 멤버에게 통지한다. (41장)

### 47.2 분할 결제·가중 분담 (확장 포인트)

- MVP는 18.2의 균등 분할을 기본으로 한다.
- 다중 결제자(분할 결제)와 참여자별 가중 분담은 확장 포인트로만 두고, 도입 시점은 추후 결정한다.

## 48. 부록: 통화 최소단위·표시 포맷

18.2.1의 정수 연산과 44.1의 표시 포맷은 아래 통화별 정의를 따른다. 기본값은 ISO 4217 exponent를 따르되, 일부 통화는 실무 관행에 맞춰 별도 취급한다.

| 통화 | 국가/지역 | 심볼 | ISO 4217 소수 자릿수 | 서비스 최소단위 | 비고 |
|---|---|---|---|---|---|
| KRW | 대한민국 | ₩ (원) | 0 | 1 (1원) | 정산 통화 기본값 |
| JPY | 일본 | ¥ | 0 | 1 (1엔) | |
| VND | 베트남 | ₫ | 0 | 1 (1동) | |
| TWD | 대만 | NT$ | 2 | 1 (정수) | ISO상 2자리이나 실무 정수 거래 관행에 따라 정수 취급 |
| USD | 미국 | $ | 2 | 0.01 (1센트) | |
| EUR | 유로존 | € | 2 | 0.01 | |
| THB | 태국 | ฿ | 2 | 0.01 (1사땅) | |
| GBP | 영국 | £ | 2 | 0.01 (1펜스) | |
| CHF | 스위스 | Fr | 2 | 0.01 | |

- 금액은 저장·계산 시 최소단위 정수로 다룬다. (예: USD는 센트 단위 정수, KRW는 원 단위 정수)
- 표에 없는 통화는 기본적으로 ISO 4217 exponent를 따른다.
- 환율은 금액이 아니므로 이 표의 최소단위를 적용하지 않으며, 소수점 6자리 권장 정밀도로 저장한다. (14.3)
- 자리구분 기호·소수점 기호·심볼 위치는 표시 로케일을 따른다. (44.2)
- 이 표는 `currencies` 룩업 테이블(code·iso_exponent·minor_unit·symbol)로 구체화하며, 모든 통화 컬럼(`*_currency`)이 이를 FK로 참조해 무결성을 강제한다. (SSOT)
